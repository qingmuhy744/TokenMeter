# 规格说明：模型套餐（Plan Suites）与多维性能矩阵大盘

## 1. 概述
TokenMeter 目前每个测试计划（Plan）都是独立的，导致管理多个模型时配置重复（API Key/Base URL 需要反复填写）。同时，主界面缺乏多维度的统计对比（如白天/夜间性能差异、长期趋势统计）。

本设计旨在引入“模型套餐（Suite）”概念实现配置继承与顺序测试，并重构前端大盘为高信息密度的彩色热力矩阵。

## 2. 用户场景
- **批量管理：** 用户创建一个“OpenAI 套餐”，填写一次 API Key，然后批量录入 10 个模型名，系统自动顺序测试。
- **性能评估：** 用户通过大盘一眼看出哪些模型在“白天（高峰期）”发生了显著劣化。
- **深度分析：** 点击某个模型，查看包含 HTTP TTFB、思考时间、首字时间等所有微观指标的原始记录。

## 3. 系统架构

### 3.1 数据模型（Backend）
对 `TokenPlan` 表进行升级：
- **`parent_id` (FK, nullable):** 自引用。
    - `NULL`: 该项为“套餐（Suite）”或独立计划。存储通用配置。
    - `NOT NULL`: 该项为“子模型”。继承父级的配置，存储模型名和频率倍率。
- **`multiplier` (Float, default=1.0):** 仅子模型有效。决定其实际触发频率（`套餐基础频率 / multiplier`）。

### 3.2 调度逻辑（Scheduler）
- **心跳机制：** APScheduler 仅为 `parent_id` 为空的项注册任务。
- **批次执行：** 
    1. 当套餐任务触发时，检索其所有活跃子模型。
    2. 根据 `multiplier` 判定哪些模型在本次心跳应当执行。
    3. **顺序队列：** 将到期模型排队，循环执行。
    4. **随机间隔：** 每次子任务执行后，执行 `await asyncio.sleep(random.uniform(2, 10))` 以规避并发限制。

### 3.3 前端大盘（Advanced Dashboard）

#### A. 核心状态过滤
- **时段维度：** 全部 (All) | 白天 (Day, 08:00-20:00) | 夜间 (Night, 20:00-08:00)。
- **时间窗口：** 1天 | 7天 | 30天。
- **注：** 由前端根据浏览器本地时区对数据点进行动态归类计算。

#### B. 彩色热力矩阵 (Status Table)
- **级联显示：** 名字显示为 `套餐名 > 模型名`。
- **指标着色：** 效仿 `@report.html`，对 TTFT、TPS、劣化率等单元格填充 `h1-h5` 背景色。
- **层次化指标：**
    - **核心（固定可见）：** TTFT, TPS, Status, 劣化率。
    - **详细（横向滚动）：** TTFB, TTFR, Think Duration, TPS-Content, Ping, Tokens 等。

#### C. 模型详情页 (Drill-down)
- **核心趋势图：** 仅展示 TTFT 和 TPS 的折线图，背景带时段阴影。
- **完整原始表：** 包含该模型所有详细时间指标的记录清单。

## 4. 接口设计
- `GET /api/plans/suites`: 返回套餐及其子模型的嵌套结构。
- `GET /api/results/matrix`: 返回按模型、按时段聚合后的均值/劣化率统计包。

## 5. 预估逻辑
- 前端在配置界面提供实时计算器：
    - `每日请求数 = Σ (24 * 60 * multiplier) / interval`
    - `每日预估 Token = 每日请求数 * max_tokens`

## 6. 成功标准
- [ ] 支持一个 Plan 包含多个模型。
- [ ] 套餐下的模型按顺序（非并发）测试，且带随机间隔。
- [ ] 主界面不再有全局趋势图，改为高性能彩色矩阵。
- [ ] 支持 1/7/30d 的昼夜对比统计。
- [ ] 所有的详细时间点指标均在详情页展示。
