# TokenMeter 安全与模型治理整改规格

**日期：** 2026-05-09

## 目标

对 TokenMeter 做一轮系统性整改，优先修复安全漏洞、计划模型语义错误、公开页面断链、调度不一致、测速统计口径不清和明显的代码债。整改按步骤执行，每一步结束后都必须运行对应回归测试，再进入下一步。

## 范围

本轮整改覆盖以下系统：

- 后端鉴权与 Session 安全
- 计划模型、继承、父子关系与调度语义
- 公开页面 API 与前端展示链路
- 测速统计结果的字段语义和实现口径
- 关键查询性能与部署默认值
- 与上述变更直接相关的测试

本轮不做无关 UI 美化，不做大规模目录重构，不做数据库架构重写。

## 成功标准

### 1. 安全面

- `custom_banner` 不再允许游客执行任意脚本。
- 前端不再拿到原始 `api_key` 和继承后的完整密钥。
- Session Cookie 显式启用安全参数，管理写操作具备基本的跨站风险防护能力。
- `api_base` 校验与实际请求执行之间的 SSRF 风险面被明显收窄。

### 2. 计划模型面

- 后端禁止计划自引用和环引用。
- “套餐/suite”和“普通计划/leaf plan”在后端拥有明确判定规则。
- 手动触发与调度触发行为和新判定规则一致。
- `multiplier` 的允许范围和实际行为一致。

### 3. 公开链路面

- 公开矩阵页面不再请求不存在的接口。
- 公共详情页不再请求私有统计接口。
- 游客能稳定访问公开状态、公开矩阵、公开历史和公开计划详情页。

### 4. 调度与数据面

- 调度状态不再依赖脆弱的错误模型语义。
- 导入逻辑不再出现“已写库但接口报失败”的假失败状态。
- 高频查询具备最低限度的索引支撑。

### 5. 测速统计面

- 结果字段名与实际语义一致，估算值与真实值明确区分。
- 调试字段、调试日志和用户可见字段边界清楚。

## 约束

- 尽量沿用现有 FastAPI、SQLAlchemy、React 结构，不做无关大重构。
- 优先通过新增测试锁定回归风险。
- 每一步实现完成后必须立即运行该步对应测试。
- 可以调整前后端契约，但前后端必须同时改完并补测试。

## 设计方案

### 一、后端安全策略

#### 1. `custom_banner`

本轮不保留任意 HTML 渲染能力，改为安全文本或受限展示内容。原因：

- 当前系统没有成熟的 HTML 白名单清洗链路。
- 公开状态页是游客入口，风险最高。
- 先降级到安全展示，比引入一套不完整 sanitizer 更稳妥。

前端不再使用 `dangerouslySetInnerHTML`。后端保持字符串存储即可，但仅按文本展示。

#### 2. API Key 返回策略

`PlanResponse` 不再返回：

- `api_key`
- `effective_api_key`

改为返回布尔或掩码信息，例如：

- `has_api_key`
- `has_effective_api_key`

前端编辑计划时不再依赖回显原始密钥。更新逻辑继续采用“仅在用户重新填写时才发送新 key”的模式。

#### 3. Session / Cookie

SessionMiddleware 显式配置：

- `same_site`
- `https_only`
- 自定义 cookie 名称

如果当前前后端同源部署，则优先使用较严格的 `SameSite=Lax`。同时为后续写操作预留 CSRF 防护扩展点；本轮至少完成 Cookie 安全属性收紧。

#### 4. SSRF 缩口

保留现有字面地址校验，同时增加更严格的执行前校验策略：

- 限制协议为 `http/https`
- 拒绝显式回环/内网/保留地址
- 对域名做解析结果校验，拒绝解析到内网/回环/链路本地
- 关闭或限制危险重定向穿透

本轮目标不是做完美 SSRF 沙箱，而是让风险从“容易绕过”降到“需要更高成本绕过”。

### 二、计划模型与调度语义

#### 1. 根计划类型判定

当前系统把所有 `parent_id is null` 的计划都当作 suite，这是错误的。

整改后语义：

- `suite`：根节点，且拥有子计划
- `standalone plan`：根节点，但没有子计划
- `child plan`：有父计划

执行规则：

- 手动触发 `suite`：顺序触发所有子计划
- 手动触发 `standalone plan`：直接执行单计划测速
- 手动触发 `child plan`：直接执行单计划测速
- 调度同步只为 `suite` 和 `standalone plan` 注册合理任务，不再混用错误语义

#### 2. 父子关系校验

后端新增校验：

- 不能把自己设为自己的父节点
- 不能形成环
- 不能把子计划提升为父节点后形成闭环

该校验应同时覆盖创建和更新。

#### 3. `multiplier`

当前实现本质是降采样，不支持真正的“高于父频率”。整改后要么：

- 明确限制 `0 < multiplier <= 1`

要么扩展完整高频调度。

本轮选择第一种，原因是：

- 和现有调度器实现一致
- 改动更小
- 便于测试和文案统一

前端文案改为“子计划执行比例/降采样倍率”或等价表达，避免误导用户。

#### 4. `suite_ticks`

本轮不引入复杂持久化调度状态，但至少做到：

- 状态清理与 job 生命周期一致
- 不再依赖错误的 suite 定义
- 测试覆盖重置和清理行为

如果实现成本可控，可顺带把其生命周期收敛到调度同步逻辑里。

### 三、公开页面链路

#### 1. 公开矩阵

新增 `/api/public/matrix`，返回与私有矩阵兼容的只读脱敏数据结构。

原因：

- 公开状态页已集成矩阵交互
- 让公共/私有矩阵共用展示组件，风险最小

#### 2. 公共计划详情与历史

公共详情页不得再请求私有 stats。两种可选方式里，本轮选择：

- 公共详情页隐藏私有 stats 卡片
- 历史列表继续使用公共结果接口

这样实现简单、权限边界清晰，不引入额外公开统计面。

### 四、测速结果字段收敛

本轮不重写整个测速器，但会修正最容易误导的数据输出：

- 明确 `content_tokens/thinking_tokens/tps_content` 是估算值，或停止在公开页/管理页突出展示它们
- 清理生产调试日志
- 收敛 `debug_chunks`、`ping_samples` 的用途边界

如果字段保留在数据库中，可以继续存储，但不应被前端当作高可信主指标。

### 五、事务与索引

#### 1. 导入事务边界

导入流程应保证：

- 数据提交成功且调度同步成功，接口才返回成功
- 若调度同步失败，错误语义必须准确，不能伪装成“数据库导入失败”

若无法做到同事务，则至少拆分错误处理，让返回值反映真实状态。

#### 2. 索引

本轮增加最关键索引：

- `test_results(plan_id, created_at)`
- `test_results(created_at)`
- `token_plans(parent_id)`
- `token_plans(is_active)`

如果 Alembic autogenerate 不可靠，手写迁移。

## 测试策略

每一类整改都先补或修测试，再落实现。至少包括：

- `custom_banner` 安全文本展示测试
- 计划响应不再暴露原始密钥的 API 测试
- 公共矩阵接口测试
- 公共详情页不请求私有 stats 的前端/契约测试
- 父子关系防环测试
- suite / standalone / child 三类触发行为测试
- `multiplier` 取值校验测试
- 登录限流修复测试
- 索引迁移可执行性验证

## 分步执行策略

按以下顺序执行，每步结束立即回归：

1. 安全面：banner、密钥响应、Session 参数
2. 公开链路：public matrix、public detail/history
3. 计划模型：suite 判定、环校验、multiplier 约束
4. 调度与导入一致性
5. 索引与部署默认值
6. 测速口径与调试残留清理

每一步都应具备独立可验证性，且不依赖后续步骤才能成立。

## 风险与取舍

- `custom_banner` 从 HTML 降级为文本，可能影响现有展示灵活性，但这是换安全性的明确取舍。
- `api_key` 不再回显后，前端编辑体验会改变，但这是必要的安全收敛。
- `multiplier` 收紧到 `<= 1` 可能影响少量已有数据，需要前后端一起兜底校验和提示。
- 公开矩阵新增接口会扩大公开读面，但数据结构保持脱敏，不包含任何管理敏感信息。
